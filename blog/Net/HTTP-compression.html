# HTTP-compression

在HTTP优化中，可以通过减小HTTP响应的大小来减少响应时间。为实现减小HTTP响应的大小，可以对响应体进行压缩，通常使用的压缩方法是Gzip。

# 启用Gzip

客户端在发起请求时，在header添加Accept-Encoding头，来声明一系列的可以支持压缩模式。
Accept-Encoding: gzip, deflate：客户端可支持的压缩模式：gzip、deflate。

服务端接受到请求后，寻找Accept-Encoding中自身可支持的编码，而后进行响应，并且在响应中设置Content-Encoding首部提供了实际采用的压缩模式。 Content-Encoding: gzip，这里表明服务端使用gzip模式进行压缩编码。

客户端得到响应结果，识别响应头Content-Encoding: gzip，而后对响应体进行gzip解压，再做下一步处理。

对于浏览器来说，解压gzip等操作由浏览器提供，并自动处理。在web开发的过程中，我们并不需要对响应压缩进行处理，浏览器会替我们搞定。

# 压缩的优缺点

优点：减小响应体，降低数据传输时间，降低流量。 缺点：压缩需要消耗服务端一定的计算性能和计算时间。

# 压缩什么

一般建议对数据尽可能地进行压缩。不过对于特定类型的文件来说，比如jpeg图片文件，已经是进行过压缩的了。有时候再次进行额外的压缩无助于负载体积的减小，反而有可能会使其增大。
所以需要避免对这种特定类型文件的压缩，避免收益小于消耗的情况发生。

# 压缩等级

gzip的压缩可分为1-9，9个等级，等级越高压缩率越高。在HTTP中，并不是压缩的越狠越好，更高的压缩率也需要更高的计算性能和时间消耗，所以要综合考虑。
在nodejs中的zlib库中，zlib.Z_DEFAULT_COMPRESSION = 6，也就是6压缩等级。compression库中依赖zlib压缩，默认也是使用的这个压缩等级。 在compression-webpack-plugin中情况有所不同，参数compressionOptions.level用于指定压缩等级，它的默认值是9，也就是最高的压缩等级。

为何会出现这样的状况，我认为是使用情况不同。在服务端，需要综合考虑压缩计算性能及时间的消耗。压缩率太高，计算时间长，性能消耗大；压缩率太低，计算时间短，但压缩结果与未压缩相差不大，显得压缩没有必要。所以了一个中间结果，也就是等级6。
而compression-webpack-plugin用于构建阶段，并不需要考虑压缩时间的问题，即使长一点也没关系。所以尽可能的将文件压缩小，也就默认使用最高的压缩等级。

# 针对代理服务器的处理

当使用代理服务器时，需要在被代理服务中设置Vary: Accept-Encoding头，告诉代理服务器缓存两种版本的资源：压缩和非压缩。这有助于避免一些公共代理不能正确地检测Content-Encoding标头的问题。

# 在Express项目中开启gzip

首先需要了解两个库：compression,express-static-gzip。

# 默认情况

在Express中，可以直接使用compression直接开启gzip。

const express = require('express')
const compression = require('compression')
const app = express()
app.use(compression({ filter: shouldCompress }))
// ...

# 存在预压缩文件的情况

举一个例子来说，假如前端使用vue/react来开发，服务端采用nodejs，并且不采用分离部署的方式，而是直接将前端构建结果丢到express的静态资源中。在前端的构建过程中，已经将gzip生成好了。

index.html
main.js
main.js.gz
style.css
style.css.gz

此时再使用compression就显得有些傻了，因为无论请求的内容是否已经存在预编译gzip，compression都不会关心，它只会勤勤恳恳的将内容压缩。比如我们请求main.js，实际上已经存在main.js.gz的预压缩文件，服务器把它直接返回就好了。但compression不会这么做，而是消耗时间和性能，将main.js压缩。

为了让我们的预压缩资源生效，我们需要对此做出改变：当请求资源为静态资源，且有预压缩文件时，返回预压缩文件；否则，压缩返回。

这里就需要使用到express-static-gzip库，来对静态文件进行处理：

const express = require('express')
const expressStaticGzip = require('express-static-gzip')
const compression = require('compression')

const app = express()
// ...
// Tip：此代码需移除。
// app.use(express.static(path.join(__dirname, 'public')))
// ...

// 针对有预编译的静态资源，直接返回预编译结果
app.use('/', expressStaticGzip(path.join(__dirname, 'public'), {
  index: false,
  serveStatic: {
    index: 'index.html',
  },
}))
// 其他情况压缩
app.use(compression())

// ...

需要注意的是，如果按照此方法使用，一定要将expressStaticGzip中options.index设置为false，否则expressStaticGzip在需要下坠的情况会错误的执行next()。原因可查看相关issue
